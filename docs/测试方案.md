# Sandbox Operator 测试方案

本文用于验证当前 `sandbox-operator` 的核心行为：

- Kubernetes CR 创建、更新、删除是否正确同步到 Sandbox OpenAPI。
- OpenAPI/控制台侧创建、更新、删除是否能被 Poller 同步回 CR。
- CR 字段映射是否完整，是否存在漏字段或单位/枚举错误。
- 更新时是否只发送必要字段。
- OpenAPI 调用失败时，webhook、reconciler、poller 是否按预期处理。

当前实现约束：

- OpenAPI 是权威源。
- 模板 ID 和沙箱 ID 只维护在受保护 annotation 中：
  - `sandbox.kce.ksyun.com/template-id`
  - `sandbox.kce.ksyun.com/sandbox-id`
  - `sandbox.kce.ksyun.com/sandbox-ids`
- `status` 不再维护模板 ID、沙箱 ID。
- 删除 CR 一定删除对应 OpenAPI 资源；没有 `deletionPolicy`。
- 创建由 mutating webhook 调 OpenAPI 并注入 ID annotation。
- 更新由 validating webhook 调 OpenAPI，随后 reconciler/poller 再 Get OpenAPI 回写 CR。
- 控制台/OpenAPI 直接创建或修改的资源由 Poller adoption 或同步回 CR。

## 1. 测试环境准备

### 1.1 前置条件

准备一个可访问 Sandbox OpenAPI 的 Kubernetes 集群，并确认：

```bash
kubectl version
helm version
kubectl get nodes
```

部署 operator：

```bash
helm upgrade --install sandbox-operator charts/sandbox-operator \
  -n sandbox-operator-system \
  --create-namespace
```

确认组件运行：

```bash
kubectl get pods -n sandbox-operator-system
kubectl get validatingwebhookconfiguration sandbox-operator-validating-webhook
kubectl get mutatingwebhookconfiguration sandbox-operator-mutating-webhook
kubectl get crd sandboxtemplates.sandbox.kce.ksyun.com
kubectl get crd sandboxes.sandbox.kce.ksyun.com
kubectl get crd sandboxclaims.sandbox.kce.ksyun.com
```

预期：

- operator Pod 为 `Running`。
- mutating webhook 和 validating webhook 均存在。
- 三个 CRD 均存在。

### 1.2 创建业务命名空间和凭据

示例命名空间：

```bash
kubectl create namespace sandbox-demo
```

创建 OpenAPI 凭据 Secret，必须与业务 CR 在同一 Namespace：

```bash
kubectl -n sandbox-demo create secret generic sandbox-openapi-credentials \
  --from-literal=accessKeyId='<AK>' \
  --from-literal=secretAccessKey='<SK>' \
  --from-literal=accountId='<ACCOUNT_ID>' \
  --from-literal=region='cn-beijing-6'
```

如要测试 KS3/KPFS/Klog/Image 凭据，创建同 Namespace Secret：

```bash
kubectl -n sandbox-demo create secret generic ks3-credential \
  --from-literal=accessKey='<KS3_AK>' \
  --from-literal=secretAccessKey='<KS3_SK>' \
  --from-literal=token=''

kubectl -n sandbox-demo create secret generic kpfs-credential \
  --from-literal=accessKey='<KPFS_AK>' \
  --from-literal=secretAccessKey='<KPFS_SK>' \
  --from-literal=token=''

kubectl -n sandbox-demo create secret generic klog-credential \
  --from-literal=accessKey='<KLOG_AK>' \
  --from-literal=secretAccessKey='<KLOG_SK>' \
  --from-literal=token=''
```

### 1.3 观察日志

测试期间建议开两个窗口：

```bash
kubectl logs -n sandbox-operator-system deploy/sandbox-operator -f
```

```bash
kubectl get sandboxtemplates,sandboxes,sandboxclaims -n sandbox-demo -w
```

关键日志：

- webhook create/update 是否调用 OpenAPI。
- poller 是否扫描到业务 Namespace。
- adoption 数量。
- sync namespace failed。
- Reconciler error。

## 2. 通用验收标准

每个测试用例至少检查以下内容：

```bash
kubectl get sandboxtemplate -n sandbox-demo -o wide
kubectl get sandbox -n sandbox-demo -o wide
kubectl get sandboxclaim -n sandbox-demo -o wide
kubectl get <resource> -n sandbox-demo <name> -o yaml
```

通用预期：

- 创建后 CR metadata annotation 中存在平台 ID。
- `status` 中没有 `templateID`、`sandboxID` 字段。
- `status.phase/rawStatus/conditions/observedGeneration` 按 OpenAPI 状态回写。
- OpenAPI 删除资源后，本地 CR 被删除。
- 删除 CR 后，OpenAPI 资源被删除。
- 用户不能手动设置、修改或删除内部 ID annotation。

检查 ID annotation：

```bash
kubectl get sandboxtemplate -n sandbox-demo <name> \
  -o jsonpath='{.metadata.annotations.sandbox\.kce\.ksyun\.com/template-id}{"\n"}'

kubectl get sandbox -n sandbox-demo <name> \
  -o jsonpath='{.metadata.annotations.sandbox\.kce\.ksyun\.com/sandbox-id}{"\n"}'
```

检查 status 不含 ID：

```bash
kubectl get sandboxtemplate -n sandbox-demo <name> -o jsonpath='{.status.templateID}{"\n"}'
kubectl get sandbox -n sandbox-demo <name> -o jsonpath='{.status.sandboxID}{"\n"}'
```

预期输出为空。

## 3. CR -> OpenAPI 创建测试

### 3.1 创建最小模板

创建文件 `template-minimal.yaml`：

```yaml
apiVersion: sandbox.kce.ksyun.com/v1alpha1
kind: SandboxTemplate
metadata:
  name: e2e-template-minimal
  namespace: sandbox-demo
spec:
  description: "e2e minimal template"
  type: Custom
  access: Private
  template:
    spec:
      image:
        source: Public
        image: "hub.kce.ksyun.com/sandbox/aio:v20260608"
      resources:
        cpu: "2"
        memory: 4096Mi
        disk: 20480Mi
      ports:
        - name: http
          containerPort: 8080
          protocol: TCP
      startCommand: "python /home/user/app.py"
      env:
        - name: APP_ENV
          value: "prod"
  pool:
    targetSize: 1
```

执行：

```bash
kubectl apply -f template-minimal.yaml
kubectl get sandboxtemplate -n sandbox-demo e2e-template-minimal -o yaml
```

预期：

- mutating webhook 调用 `CreateSandboxTemplate` 成功。
- CR 被写入并带有 `sandbox.kce.ksyun.com/template-id` annotation。
- `status.phase` 最终为 `Ready` 或 OpenAPI 对应状态。
- `spec` 中镜像、资源、端口、启动命令、环境变量、pool 均与 OpenAPI 最终值一致。
- 控制台/OpenAPI 能看到同名模板。

### 3.2 创建完整字段模板

使用 `config/samples/sandbox_v1alpha1_sample.yaml` 中的 `SandboxTemplate`，或按需创建包含以下字段的模板：

- `spec.description`
- `spec.type`
- `spec.access`
- `spec.template.spec.image.source`
- `spec.template.spec.image.image`
- `spec.template.spec.image.registryCredentialRef`
- `spec.template.spec.resources.cpu`
- `spec.template.spec.resources.memory`
- `spec.template.spec.resources.disk`
- `spec.template.spec.ports`
- `spec.template.spec.startCommand`
- `spec.template.spec.env`
- `spec.template.spec.volumes`，包含 KS3 和 KPFS
- `spec.template.spec.networkConfig`
- `spec.template.spec.skillConfig`
- `spec.template.spec.dataDisks`
- `spec.pool.targetSize`
- `spec.observability.logging`

执行：

```bash
kubectl apply -f config/samples/sandbox_v1alpha1_sample.yaml
kubectl get sandboxtemplate -n sandbox-demo custom-app -o yaml
```

预期：

- OpenAPI 创建成功。
- CR annotation 有 template-id。
- OpenAPI 侧模板字段与 CR 输入一致，或与 OpenAPI 规范化后的值一致。
- Poller/Reconciler 回写后 CR 字段不丢失。

重点核对：

| CR 字段 | OpenAPI 字段 | 预期 |
| --- | --- | --- |
| `metadata.name` | `TemplateName` | 一致 |
| `spec.description` | `Description` | 一致 |
| `spec.type` | `TemplateType` | CR 显示 `AIO/BROWSER/CODE/CUSTOM`，发 OpenAPI 用 `AIO/browser/code/custom` |
| `spec.access` | `TemplateCategory` | `Private/Public` 映射正确 |
| `resources.cpu` | `CPU` | 数值正确 |
| `resources.memory` | `Memory` | 单位转换正确 |
| `resources.disk` | `KecConfig.SystemDiskSize` 或等价字段 | 单位转换正确 |
| `env` | `Envs` | 数组顺序和值正确 |
| `volumes.KS3` | `KS3MountConfig` | enable、mountPath、bucket、path、readOnly、凭据正确 |
| `volumes.KPFS` | `KPFSMountConfig` | enable、mountPath、fileSystem、path、readOnly、凭据正确 |
| `pool.targetSize` | `PreheatConfig` / `TargetPoolSize` | 一致 |
| `observability.logging` | `KlogConfig` | project、endpoint、pool、rules、凭据正确 |

### 3.3 创建沙箱

使用模板名创建沙箱：

```yaml
apiVersion: sandbox.kce.ksyun.com/v1alpha1
kind: Sandbox
metadata:
  name: e2e-sandbox
  namespace: sandbox-demo
spec:
  name: e2e-sandbox
  templateRef:
    name: e2e-template-minimal
  timeoutSeconds: 1800
```

执行：

```bash
kubectl apply -f sandbox.yaml
kubectl get sandbox -n sandbox-demo e2e-sandbox -o yaml
```

预期：

- mutating webhook 读取模板 CR 的 template-id annotation。
- 调用 `StartSandboxInstance` 成功。
- CR 写入 `sandbox.kce.ksyun.com/sandbox-id` annotation。
- `status.phase` 最终为 `Running` 或 OpenAPI 对应状态。
- `status.timeoutSeconds` 与 OpenAPI 一致。
- `status.endpoint`、`status.token` 按 OpenAPI 返回或后续同步结果呈现。

### 3.4 创建 SandboxClaim

```yaml
apiVersion: sandbox.kce.ksyun.com/v1alpha1
kind: SandboxClaim
metadata:
  name: e2e-claim
  namespace: sandbox-demo
spec:
  replicas: 2
  templateRef:
    name: e2e-template-minimal
  timeoutSeconds: 1800
```

执行：

```bash
kubectl apply -f claim.yaml
kubectl get sandboxclaim -n sandbox-demo e2e-claim -o yaml
kubectl get sandbox -n sandbox-demo -l sandbox.kce.ksyun.com/claim=e2e-claim -o yaml
```

预期：

- Claim annotation 有 `template-id` 和 `sandbox-ids`。
- Reconciler 创建 `e2e-claim-0`、`e2e-claim-1` 子 Sandbox CR。
- 每个子 Sandbox 有 `sandbox-id` annotation。
- Claim status 汇总 `desired/created/ready/failed/phase`。

## 4. CR -> OpenAPI 更新测试

### 4.1 模板描述和访问类型更新

执行：

```bash
kubectl patch sandboxtemplate -n sandbox-demo e2e-template-minimal --type=merge -p '{
  "spec": {
    "description": "e2e updated description",
    "access": "Public"
  }
}'
```

预期：

- validating webhook 调用 `UpdateSandboxTemplate`。
- OpenAPI 侧 description、access 更新。
- Reconciler/Poller 后续 Get OpenAPI 并回写 CR。
- annotation 中 template-id 不变。

### 4.2 模板预热池更新

执行：

```bash
kubectl patch sandboxtemplate -n sandbox-demo e2e-template-minimal --type=merge -p '{
  "spec": {
    "pool": {
      "targetSize": 2
    }
  }
}'
```

预期：

- OpenAPI target pool size 更新为 2。
- CR `spec.pool.targetSize` 后续仍为 2。
- `status.preheat.number` 或等价字段同步为 OpenAPI 返回值。

### 4.3 模板资源更新

分别测试 CPU、Memory、Disk：

```bash
kubectl patch sandboxtemplate -n sandbox-demo e2e-template-minimal --type=merge -p '{
  "spec": {
    "template": {
      "spec": {
        "resources": {
          "cpu": "4",
          "memory": "8192Mi",
          "disk": "30720Mi"
        }
      }
    }
  }
}'
```

预期：

- CPU、Memory、Disk 均更新到 OpenAPI。
- Memory 单位转换正确，不出现 `Memory格式无效`。
- Disk 通过 KecConfig 或 OpenAPI 对应字段更新。
- OpenAPI 回写 CR 后不回退。

### 4.4 模板环境变量部分更新

执行：

```bash
kubectl patch sandboxtemplate -n sandbox-demo e2e-template-minimal --type=merge -p '{
  "spec": {
    "template": {
      "spec": {
        "env": [
          {"name": "APP_ENV", "value": "staging"},
          {"name": "LOG_LEVEL", "value": "debug"}
        ]
      }
    }
  }
}'
```

预期：

- OpenAPI `Envs` 更新。
- 未修改字段不被清空。
- CR 回写后 env 保持为 OpenAPI 最终值。

### 4.5 KS3/KPFS 挂载删除和新增

删除全部 KS3/KPFS：

```bash
kubectl patch sandboxtemplate -n sandbox-demo custom-app --type=json -p='[
  {"op":"remove","path":"/spec/template/spec/volumes"}
]'
```

预期：

- webhook 调 `UpdateSandboxTemplate` 时对 KS3/KPFS 显式发送 disabled 状态。
- OpenAPI 侧挂载被删除。
- 下一轮同步后 CR 不再恢复旧挂载。

再新增一个 KS3 挂载，必须带 credentialRef：

```bash
kubectl patch sandboxtemplate -n sandbox-demo custom-app --type=merge -p '{
  "spec": {
    "template": {
      "spec": {
        "volumes": [
          {
            "name": "ks3-new",
            "type": "KS3",
            "mountPath": "/mnt/ks3/new",
            "readOnly": true,
            "ks3": {
              "bucket": "bucket-a",
              "path": "/new",
              "credentialRef": {"name": "ks3-credential"}
            }
          }
        ]
      }
    }
  }
}'
```

预期：

- OpenAPI 挂载新增成功。
- 如果缺少 `credentialRef`，webhook 拒绝。

### 4.6 沙箱 timeout 更新

执行：

```bash
kubectl patch sandbox -n sandbox-demo e2e-sandbox --type=merge -p '{
  "spec": {
    "timeoutSeconds": 3600
  }
}'
```

预期：

- validating webhook 调 `UpdateSandboxInstance`。
- OpenAPI 侧 timeout 更新。
- CR `status.timeoutSeconds` 后续为 3600。
- sandbox-id annotation 不变。

### 4.7 部分更新验证

目标：确认模板更新只发送变化字段，不把未修改字段清空。

测试方法：

1. 在 OpenAPI 或控制台记录模板完整详情。
2. 只 patch 一个字段，例如 `spec.description`。
3. 查看 OpenAPI 请求日志或服务端审计日志。
4. 查看 OpenAPI 最终详情。

预期：

- `UpdateSandboxTemplate` 请求只包含 `TemplateId` 和变化字段。
- 未变化字段，例如 image、env、ports、mounts、network、logging、pool，不被覆盖为空。
- 如果更新 disk/dataDisks，允许请求补齐 OpenAPI 要求的 KecConfig 必填字段。

当前需要重点验证的 diff 分组：

| 修改内容 | 预期发送 |
| --- | --- |
| description | 只发 description |
| access | 只发 category/access |
| type | 只发 type |
| image | 只发 imageConfig |
| ports | 只发 ports |
| startCommand | 只发 command |
| env | 只发 envs |
| resources.cpu/memory | 发 cpu/memory |
| disk/dataDisks | 发 KecConfig，并从 remote 补齐必要字段 |
| networkConfig | 只发 networkConfig |
| skillConfig | 只发 skillConfig |
| KS3/KPFS volumes | 同时发 KS3/KPFS 当前目标状态和凭据 |
| observability.logging | 只发 klogConfig |
| pool.targetSize | 只发 preheat/target pool |

## 5. CR 删除测试

### 5.1 删除沙箱

执行：

```bash
SANDBOX_ID=$(kubectl get sandbox -n sandbox-demo e2e-sandbox -o jsonpath='{.metadata.annotations.sandbox\.kce\.ksyun\.com/sandbox-id}')
kubectl delete sandbox -n sandbox-demo e2e-sandbox
```

预期：

- CR 进入 deleting 后被 finalizer 处理。
- operator 调 `DeleteSandboxInstance`。
- OpenAPI 查询该 sandbox ID 返回不存在。
- CR 最终从 Kubernetes 删除。
- Poller 不会再次 adoption 该沙箱。

### 5.2 删除模板

执行：

```bash
TEMPLATE_ID=$(kubectl get sandboxtemplate -n sandbox-demo e2e-template-minimal -o jsonpath='{.metadata.annotations.sandbox\.kce\.ksyun\.com/template-id}')
kubectl delete sandboxtemplate -n sandbox-demo e2e-template-minimal
```

预期：

- operator 调 `DeleteSandboxTemplate`。
- OpenAPI 模板被删除。
- CR 最终删除。

注意：

- 如果 OpenAPI 因模板仍有实例等原因拒绝删除，CR 会保留 finalizer 并停留在 terminating。
- 清理依赖资源后，reconciler 应重试并最终删除。

### 5.3 删除 SandboxClaim

执行：

```bash
kubectl delete sandboxclaim -n sandbox-demo e2e-claim
```

预期：

- Claim finalizer 删除子 Sandbox CR。
- 子 Sandbox finalizer 删除对应 OpenAPI 沙箱实例。
- Claim 和子 Sandbox 最终都被删除。

## 6. OpenAPI -> CR 同步测试

### 6.1 OpenAPI 创建模板后 adoption

在控制台或 OpenAPI 创建一个模板，例如名称 `console-template-e2e`。

等待一个 poll 周期：

```bash
kubectl get sandboxtemplate -n sandbox-demo
kubectl get sandboxtemplate -n sandbox-demo console-template-e2e -o yaml
```

预期：

- Poller 扫描到 OpenAPI 凭据所在 Namespace。
- 创建对应 `SandboxTemplate` CR。
- 如果 OpenAPI 模板名是合法 Kubernetes 名称，CR 名称使用模板名。
- 如果模板名不合法，CR 名称使用模板 ID。
- CR annotation 有 template-id。
- CR label 有 `sandbox.kce.ksyun.com/adopted: "true"`。
- CR spec/status 与 OpenAPI 详情一致。

### 6.2 OpenAPI 修改模板后同步回 CR

在控制台修改模板：

- description
- access
- env
- resources
- pool target size
- KS3/KPFS 挂载

等待 poll：

```bash
kubectl get sandboxtemplate -n sandbox-demo console-template-e2e -o yaml
```

预期：

- CR `spec` 被反写为 OpenAPI 最终值。
- `status.externalUpdatedAt` 或 `status.updatedAt` 更新。
- annotation template-id 不变。

### 6.3 OpenAPI 删除模板后删除 CR

在控制台或 OpenAPI 删除模板。

等待 poll：

```bash
kubectl get sandboxtemplate -n sandbox-demo console-template-e2e
```

预期：

- 本地 CR 被删除。

### 6.4 OpenAPI 创建沙箱后 adoption

在控制台或 OpenAPI 启动沙箱实例。

等待 poll：

```bash
kubectl get sandbox -n sandbox-demo
kubectl get sandbox -n sandbox-demo <adopted-name> -o yaml
```

预期：

- Poller 创建对应 `Sandbox` CR。
- CR annotation 有 sandbox-id。
- 如果 OpenAPI 返回合法沙箱名，CR 名称使用沙箱名；否则使用 sandbox ID。
- CR label 有 `sandbox.kce.ksyun.com/adopted: "true"`。
- `spec.name`、`spec.templateRef.id`、`spec.timeoutSeconds`、`status.phase` 等同步正确。

### 6.5 OpenAPI 修改沙箱 timeout 后同步回 CR

在控制台或 OpenAPI 修改沙箱 timeout。

等待 poll：

```bash
kubectl get sandbox -n sandbox-demo <name> -o yaml
```

预期：

- `spec.timeoutSeconds` 被 OpenAPI 结果覆盖。
- `status.timeoutSeconds` 同步更新。

### 6.6 OpenAPI 删除沙箱后删除 CR

在控制台或 OpenAPI 删除沙箱。

等待 poll：

```bash
kubectl get sandbox -n sandbox-demo <name>
```

预期：

- 本地 CR 被删除。

### 6.7 不同状态沙箱同步

准备不同状态沙箱：

- `STARTING`
- `RUNNING`
- `PAUSED`
- `RESUMING`
- `FAILED`
- `UNHEALTHY`
- `KILLING`

等待 poll 后检查：

```bash
kubectl get sandbox -n sandbox-demo
```

预期：

- Poller 能按状态列表拉取并去重。
- `PAUSED` 沙箱应被同步。如果未同步，优先检查 OpenAPI `GetSandboxInstanceList` 是否支持 State 参数并返回该状态。

## 7. Annotation 保护测试

### 7.1 创建时禁止用户指定 ID annotation

```yaml
apiVersion: sandbox.kce.ksyun.com/v1alpha1
kind: SandboxTemplate
metadata:
  name: invalid-template-id
  namespace: sandbox-demo
  annotations:
    sandbox.kce.ksyun.com/template-id: fake-id

  description: "invalid"
  type: Custom
  access: Private
```

执行：

```bash
kubectl apply -f invalid-template-id.yaml
```

预期：

- webhook 拒绝。
- 错误信息包含 annotation 由 operator 管理，用户不能设置。

### 7.2 更新时禁止修改 ID annotation

```bash
kubectl annotate sandboxtemplate -n sandbox-demo custom-app \
  sandbox.kce.ksyun.com/template-id=fake --overwrite
```

预期：

- webhook 拒绝。
- 原 annotation 未变化。

```bash
kubectl annotate sandbox -n sandbox-demo <name> \
  sandbox.kce.ksyun.com/sandbox-id-
```

预期：

- webhook 拒绝。
- 原 annotation 仍存在。

## 8. 失败处理测试

### 8.1 OpenAPI 凭据不存在

创建一个没有 `sandbox-openapi-credentials` 的 Namespace：

```bash
kubectl create namespace sandbox-no-cred
kubectl apply -n sandbox-no-cred -f template-minimal.yaml
```

预期：

- webhook 拒绝创建。
- 错误指向 OpenAPI credential Secret 不存在。
- Poller 日志显示跳过没有 OpenAPI 凭据的 Namespace。

### 8.2 OpenAPI 凭据字段缺失

创建缺少 `secretAccessKey` 或 `region` 的 Secret 后创建 CR。

预期：

- webhook 拒绝。
- 错误指向 Secret 缺少必填字段。

### 8.3 运行时凭据缺失

创建包含 KS3/KPFS/Klog/Image 凭据引用的模板，但不创建对应 Secret。

预期：

- webhook 拒绝创建或更新。
- 不调用 OpenAPI 或 OpenAPI 不产生资源。

### 8.4 OpenAPI 创建失败

用非法参数触发 OpenAPI 创建失败，例如：

- 不存在的镜像。
- 非法资源规格。
- 无权限的 KS3/KPFS 凭据。

预期：

- mutating webhook 拒绝 admission。
- CR 不落库。
- OpenAPI 不应产生资源；如果 OpenAPI 部分创建成功，需要通过 Poller adoption 或人工清理发现。

检查：

```bash
kubectl get sandboxtemplate -n sandbox-demo <name>
kubectl logs -n sandbox-operator-system deploy/sandbox-operator --tail=100
```

### 8.5 OpenAPI 更新失败

对已存在模板执行非法更新，例如删除 KS3 凭据但保留 KS3 挂载：

```bash
kubectl patch sandboxtemplate -n sandbox-demo custom-app --type=json -p='[
  {"op":"remove","path":"/spec/template/spec/volumes/0/ks3/credentialRef"}
]'
```

预期：

- validating webhook 拒绝。
- Kubernetes CR spec 不被更新。
- OpenAPI 侧不变化。

### 8.6 OpenAPI 删除失败

构造模板仍有关联实例、不允许删除的场景，然后删除模板 CR：

```bash
kubectl delete sandboxtemplate -n sandbox-demo <template>
```

预期：

- CR 停留在 terminating。
- finalizer 未移除。
- operator 日志有删除失败原因。
- 删除依赖实例后，reconciler 重试并最终删除 CR。

检查：

```bash
kubectl get sandboxtemplate -n sandbox-demo <template> -o yaml
kubectl logs -n sandbox-operator-system deploy/sandbox-operator --tail=100
```

### 8.7 Poller OpenAPI List/Get 失败

临时使用错误 AK/SK 或 OpenAPI endpoint，观察 poller：

预期：

- 已有 CR 不应被误删。
- 日志出现 `sync namespace failed`。
- 恢复配置后下一轮继续同步。

## 10. 并发与一致性测试

### 10.1 CR 更新与 Poller 同时同步

操作：

1. 缩短 `POLL_INTERVAL`，例如 5s。
2. 连续 patch 模板 description 或 pool targetSize。
3. 同时在控制台修改同一字段。

预期：

- 不出现 panic。
- Kubernetes conflict 被忽略或下一轮收敛。
- 最终 CR 与 OpenAPI 最终状态一致。

### 10.2 多副本 leader election

如果要测试多副本：

```bash
kubectl scale deploy/sandbox-operator -n sandbox-operator-system --replicas=2
kubectl logs -n sandbox-operator-system -l app.kubernetes.io/name=sandbox-operator --tail=100
```

预期：

- 多个 Pod 都能服务 webhook。
- 只有 leader 执行 reconciler/poller。
- 不出现重复 adoption 或重复删除。
