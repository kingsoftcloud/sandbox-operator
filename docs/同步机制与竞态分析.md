# Sandbox Operator 同步机制与竞态分析

本文说明当前 `sandbox-operator` 的同步链路、职责边界和可能存在的竞态问题。当前实现以 Sandbox OpenAPI 为权威源：用户通过 CR 发起的创建、更新、删除会先调用 OpenAPI；控制台或其他客户端直接修改 OpenAPI 后，operator 会把可反推的字段同步回 CR。

## 1. 同步机制总览

| 同步方向 | 入口 | 触发方式 | 主要职责 |
| --- | --- | --- | --- |
| CR -> OpenAPI | Mutating/Validating Webhook | 用户创建或更新 CR 时同步触发 | 创建由 mutating webhook 调 OpenAPI 并注入内部 annotation；更新由 validating webhook 调 OpenAPI 并校验。 |
| OpenAPI -> CR | Poller | 周期性轮询 | 扫描有 OpenAPI 凭据的命名空间，List/Get OpenAPI 资源，创建或更新本地 CR。 |
| OpenAPI -> CR | Reconciler | CR 事件触发 | 绑定 webhook 调用返回的 ID，从 OpenAPI Get 详情并回写 `spec`/`status`。 |
| CR 删除 -> OpenAPI 删除 | Reconciler finalizer | 用户删除 CR 时触发 | 删除对应 OpenAPI 资源。 |
| SandboxClaim -> Sandbox CR | SandboxClaimReconciler | Claim 创建后的事件触发 | 根据 webhook 启动实例后记录的 ID 创建子 Sandbox CR，并汇总 Claim 状态。 |

当前不是单纯的“reconciler 写 status、webhook 写 OpenAPI”的模型。Reconciler 和 Poller 都会在从 OpenAPI 获取到真实状态后，把可反推的 `spec` 字段反写到 CR，避免控制台修改后 CR 展示不一致。

## 2. 命名空间和凭据边界

同步按 Kubernetes Namespace 隔离。

- Poller 默认扫描所有 Namespace，但只同步能读取到 OpenAPI 凭据 Secret 的 Namespace。
- OpenAPI 凭据 Secret 默认名是 `sandbox-openapi-credentials`，也可通过 CR 的 `spec.openapiCredentialRef.name` 或 operator 配置 `DEFAULT_OPENAPI_CREDENTIAL_SECRET` 覆盖。
- OpenAPI Secret 必须在业务资源所在 Namespace 内，读取字段包括 `accessKeyId`、`secretAccessKey`、`region`，可选 `accountId`。
- 模板、沙箱引用的 KS3/KPFS/Klog/Image 等运行时凭据也只从 CR 所在 Namespace 读取。
- Poller 发现某个 Namespace 没有 OpenAPI Secret 时会跳过，不跨 Namespace 查找凭据。

凭据类字段不会从 OpenAPI 反写成 Kubernetes Secret。OpenAPI 响应里可能能看到部分凭据使用状态或非敏感配置，但 operator 不会据此创建或覆盖 Secret。对于从 OpenAPI adoption 出来的 CR，如果后续用户修改了依赖凭据的字段，例如 KS3/KPFS 挂载，则需要在 CR 中补齐同 Namespace 的 `credentialRef`，webhook 才能把完整请求发给 OpenAPI。

## 3. Webhook：CR 到 OpenAPI

Webhook 拦截 `SandboxTemplate`、`Sandbox`、`SandboxClaim` 的 CREATE/UPDATE/DELETE。

### 3.1 SandboxTemplate

创建时：

1. 校验模板基础字段，例如 `spec.type`、`spec.access`。
2. 在同 Namespace 读取 OpenAPI 凭据和运行时凭据。
3. 将 CR 转换成 `CreateSandboxTemplate` 请求。
4. OpenAPI 创建成功后，通过 mutating webhook 把返回的 `templateID` 注入内部 annotation。
5. Webhook 放行 Kubernetes 创建请求。

更新时：

1. 对比新旧 `spec`。
2. 如果 `spec` 未变化，直接放行。
3. 如果 `metadata.annotations["sandbox.kce.ksyun.com/template-id"]` 为空，拒绝更新，要求先完成 OpenAPI 绑定。
4. 根据 diff 构造 `UpdateSandboxTemplate` 请求，只发送变化字段。
5. 如果更新涉及 KS3/KPFS，剩余挂载项必须都带有有效 `credentialRef`。
6. 如果删除全部 KS3/KPFS 挂载，会显式向 OpenAPI 发送对应 `enable=false`。
7. OpenAPI 更新成功后放行 Kubernetes 更新请求。

模板类型在 CR 中按 OpenAPI 同步结果展示为 `AIO`、`BROWSER`、`CODE`、`CUSTOM`；用户提交时支持大小写不敏感，发给 OpenAPI 时使用固定形式：`AIO`、`browser`、`code`、`custom`。

### 3.2 Sandbox

创建时：

1. 校验 `spec.name` 在同 Namespace 下不重复。若 `spec.name` 为空，则使用 CR 名称作为沙箱显示名。
2. 解析 `spec.templateRef.id` 或同 Namespace 的 `spec.templateRef.name`。
3. 读取 OpenAPI 凭据和运行时凭据。
4. 调用 `StartSandboxInstance`。
5. 把返回的 `sandboxID`、`endpoint`、`token` 注入内部 annotation。
6. Webhook 放行 Kubernetes 创建请求。

更新时：

- 当前主要处理 `spec.timeoutSeconds` 变化，并调用 `UpdateSandboxInstance`。
- 其他字段变更目前会被 Kubernetes 接收，但不一定会立刻映射到 OpenAPI，随后会被 OpenAPI 同步结果覆盖。

删除时：

- Webhook 不直接删除 OpenAPI 资源，只放行请求。
- 真正删除由 Sandbox finalizer 处理。

### 3.3 SandboxClaim

创建时：

1. 校验 `spec.replicas > 0`。
2. 解析模板 ID。
3. 循环调用 `StartSandboxInstance` 启动多个沙箱。
4. 把所有返回的 `sandboxID` 注入内部 annotation。
5. Webhook 放行 Claim 创建。

Claim 的后续子 Sandbox CR 创建和状态汇总由 `SandboxClaimReconciler` 完成。

## 4. 内部 annotation 绑定记录

Webhook 不能在 admission 响应里直接修改用户提交对象的 `status`，否则既不符合 Kubernetes status 子资源的语义，也容易和 APIServer 正在持久化的对象产生资源版本冲突。因此当前实现使用 mutating webhook 在创建请求里注入内部 annotation，作为短期绑定记录。Reconciler 或 Poller 消费后会清理这些 annotation。

当前内部 annotation 包括：

```text
sandbox.kce.ksyun.com/template-id
sandbox.kce.ksyun.com/sandbox-id
sandbox.kce.ksyun.com/sandbox-ids
sandbox.kce.ksyun.com/endpoint
sandbox.kce.ksyun.com/token
```

普通用户不能在创建请求中预置这些 annotation，也不能在更新请求中修改或删除这些 annotation。operator 自己的 ServiceAccount 会绕过 webhook，用于同步回写。模板 ID、沙箱 ID 只维护在 annotation 中，不再写入 `status`。

Reconciler 和 Poller 会读取内部 annotation 中的平台 ID，再从 OpenAPI 查询详情并同步 `spec/status`。创建、更新 webhook 返回体中除 ID 外的字段不作为长期数据源，最终仍以 Get OpenAPI 的详情为准。

## 5. Reconciler：事件驱动收敛

Reconciler 处理 CR 事件，提供比 Poller 更快的本地响应。

### 5.1 Finalizer

`SandboxTemplate`、`Sandbox`、`SandboxClaim` 都会加 finalizer。

- 删除 `SandboxTemplate` 时，如果 template-id annotation 非空，则调用 `DeleteSandboxTemplate`。
- 删除 `Sandbox` 时，如果 sandbox-id annotation 非空，则调用 `DeleteSandboxInstance`。
- 删除 `SandboxClaim` 时，会删除由该 Claim 管理的子 Sandbox CR，再由子 Sandbox finalizer 处理 OpenAPI 删除。
- OpenAPI 返回 NotFound 时视为删除已完成。

### 5.2 绑定和同步

对于已存在的 CR：

1. 从内部 annotation 读取平台 ID。
2. 有平台 ID 后，使用同 Namespace OpenAPI 凭据调用 Get 接口。
3. 将 OpenAPI 详情同步回 CR `spec` 和 `status`。
4. 如果 OpenAPI 返回 NotFound，删除本地 CR。

这里删除本地 CR 的含义是：OpenAPI 是权威源，平台资源不存在时本地 CR 不再保留一个失真的对象。

### 5.3 SandboxClaim 汇总

`SandboxClaimReconciler` 会：

- 根据内部 annotation 中的 `sandboxIDs` 创建子 Sandbox CR。
- Watch 子 Sandbox 变化，并重新计算 Claim 的 `ready`、`failed`、`created` 和 `phase`。
- 不直接调用 OpenAPI 获取状态，状态来自子 Sandbox 的同步结果。

## 6. Poller：周期性 OpenAPI 到 CR 同步

Poller 是外部变更同步和兜底收敛机制。

### 6.1 Namespace 扫描

- 如果配置了 `SYNC_NAMESPACES`，只扫描指定 Namespace。
- 如果未配置，则列出所有 Namespace。
- 每个 Namespace 内尝试读取默认 OpenAPI Secret；没有 Secret 时跳过。
- Namespace 并发数由 `MAX_CONCURRENT_NAMESPACES` 控制，默认 5。

### 6.2 模板同步

对每个有 OpenAPI 凭据的 Namespace：

1. 列出本地 `SandboxTemplate` CR。
2. 对已有 CR，按 template-id annotation 调用 `GetSandboxTemplate`。
3. 将 OpenAPI 详情反写 `spec` 和 `status`。
4. 如果 OpenAPI NotFound，删除本地 CR。
5. 如果启用外部资源 adoption，调用 `GetSandboxTemplateList` 列出远端模板。
6. 对本地没有的远端模板，再调用 Get 获取详情并创建 CR。

外部模板创建 CR 时的命名规则：

- 如果 OpenAPI 模板名是合法 Kubernetes 资源名，则使用模板名。
- 否则使用完整模板 ID。
- 如果同 Namespace 已有同名 CR，则追加后缀生成唯一名称。

### 6.3 沙箱同步

对每个有 OpenAPI 凭据的 Namespace：

1. 列出本地 `Sandbox` CR。
2. 对已有 CR，按 sandbox-id annotation 调用 `GetSandboxInstance`。
3. 将 OpenAPI 详情反写 `spec` 和 `status`。
4. 如果 OpenAPI NotFound，删除本地 CR。
5. 如果启用外部资源 adoption，调用 `GetSandboxInstanceList` 分状态列出远端沙箱。
6. 对本地没有的远端沙箱创建 CR。

当前会按以下状态分别列表并去重：

```text
空状态, STARTING, RUNNING, KILLING, FAILED, UNHEALTHY, PAUSED, RESUMING
```

外部沙箱创建 CR 时的命名规则：

- 如果 OpenAPI 返回的沙箱名合法，则使用沙箱名。
- 如果沙箱名为空或不合法，则使用完整沙箱 ID。
- 如果同 Namespace 已有同名 CR，则追加后缀生成唯一名称。

### 6.4 分页

OpenAPI `PageSize` 限制为 1 到 100。Poller 会把配置值截断到最大 100，默认使用 100。分页期间如果远端列表发生变化，本轮可能出现短暂漏读或重复，重复会按平台 ID 去重，漏读依赖下一轮 Poller 收敛。

## 7. 竞态问题分析

### 7.1 Webhook 调用 OpenAPI 成功，但 CR 写入失败

时序：

1. 用户创建 CR。
2. Webhook 调用 OpenAPI 成功，平台资源已创建。
3. APIServer 持久化 CR 时失败，例如校验、etcd 或网络问题。

结果：

- 平台侧资源已经存在，但 Kubernetes 中可能没有对应 CR。
- 如果该 Namespace 后续被 Poller 扫描到，资源会被 adoption 回本地 CR。
- 如果 Poller 未启用或 Namespace 没有 OpenAPI Secret，该资源会成为平台侧孤儿资源。

当前缓解：

- Poller adoption 默认启用。
- OpenAPI 是权威源，下一轮可通过 List 发现。

残余风险：

- 在下一轮 Poller 前，用户看不到本地 CR。
- 如果 OpenAPI 创建的是部分成功结果，例如 SandboxClaim 循环启动多个沙箱时中途失败，已经启动的实例可能需要靠后续 adoption 兜底。

### 7.2 Webhook 成功，但内部 annotation 未落库

时序：

1. Webhook 调用 OpenAPI 创建成功。
2. Mutating webhook 返回的 patch 未能最终写入 etcd，例如 APIServer 持久化失败。

结果：

- 平台侧资源已经存在，但 CR 可能不存在，或 CR 存在但缺少内部 annotation。
- Poller 后续会通过 OpenAPI List 发现远端资源；但如果 CR 名称无法和远端 ID 对齐，可能创建另一个 adoption CR。

当前缓解：

- 内部 annotation 只是桥接手段，不是权威数据。
- Poller 可以兜底 adoption。

残余风险：

- 短时间内可能出现未绑定 CR。
- 极端情况下可能出现一个用户创建的空 status CR 和一个 adoption CR 并存，需要人工清理。

### 7.3 用户修改 CR 与 Poller 同步同时发生

时序：

1. 用户更新 CR。
2. Webhook 调用 OpenAPI 更新成功。
3. 同一时间 Poller 正在用稍旧的 OpenAPI 详情更新 CR。

结果：

- Kubernetes `resourceVersion` 冲突会被忽略，等待下一次事件或下一轮 Poller 收敛。
- 如果 Poller 先写入旧值，用户更新被 APIServer 接收后可能短暂被旧 OpenAPI 结果覆盖。
- 下一轮 Poller 从 OpenAPI 读取到最终状态后，会再次反写 CR。

当前策略：

- 不做用户修改与控制台修改之间的冲突检测。
- OpenAPI 负责处理并发更新，CR 最终展示 OpenAPI 真实结果。

残余风险：

- 用户可能在短时间内看到字段回跳。
- 如果 OpenAPI 返回的最终值与用户提交值不同，CR 会以 OpenAPI 返回为准。

### 7.4 控制台修改与用户修改同一字段

时序：

1. 控制台修改模板或沙箱元数据。
2. Poller 尚未同步到 CR。
3. 用户基于旧 CR 修改同一字段并提交。

结果：

- Webhook 会调用 OpenAPI 更新用户提交的字段。
- OpenAPI 底层自行处理并发，operator 不做冲突检测。
- 后续 Poller 把 OpenAPI 最终状态同步回 CR。

当前策略是 last writer wins，准确说是 OpenAPI final state wins。

残余风险：

- 控制台的修改可能被随后一次 CR 更新覆盖。
- CR 上没有字段级冲突提示，只能通过 `status.externalUpdatedAt` 观察外部更新时间。

### 7.5 Reconciler 和 Poller 同时更新同一个 CR

时序：

1. Reconciler 因 CR 事件触发 Get OpenAPI 并更新 CR。
2. Poller 同时扫描该 Namespace，也在更新同一个 CR。

结果：

- 两者都先尝试更新 `spec`，再更新 `status`。
- 如果发生 Kubernetes update conflict，当前实现通过 `ignoreConflict` 忽略错误。
- 后续事件或下一轮 Poller 会再次同步。

残余风险：

- 高频外部变更下可能持续冲突，导致某些 CR 同步延迟。
- `spec` 更新和 `status` 更新是两次 API 调用，中间可能出现 spec 已更新但 status 仍旧的短暂状态。

### 7.6 远端删除与本地删除同时发生

时序：

1. 用户删除 CR，finalizer 准备删除 OpenAPI 资源。
2. 控制台或其他客户端已经删除了同一资源。

结果：

- OpenAPI Delete 返回 NotFound 时，operator 视为删除成功并移除 finalizer。
- 如果 Poller 先发现远端 NotFound，也会删除本地 CR。

残余风险：

- 如果 OpenAPI 在短暂一致性窗口内错误返回 NotFound，本地 CR 会被删除。
- 后续如果远端资源又可见，会被 Poller adoption 为新的 CR，名称可能不同。

### 7.7 OpenAPI List 和 Get 之间资源变化

时序：

1. Poller List 得到某个远端 ID。
2. 调用 Get 详情前，该资源被删除。

结果：

- Get 返回 NotFound 时跳过 adoption 或删除已有本地 CR。

残余风险：

- 本轮同步结果可能不完整，下一轮收敛。

### 7.8 外部资源命名冲突

场景：

- 两个远端模板或沙箱在同 Namespace 下有相同的合法名称。
- 或远端名称和已有本地 CR 名冲突。

结果：

- 第一个资源使用原名。
- 后续资源使用追加后缀后的唯一名称。
- 权威绑定字段是 template-id 或 sandbox-id annotation。

残余风险：

- 用户从名称上不一定能直接看出和控制台资源的一一对应关系，需要查看 status ID。
- Kubernetes 资源名称不可变，后续控制台改名不会重命名已有 CR，只会更新 `spec.name` 或相关可反推字段。

### 7.9 凭据漂移

场景：

- 用户在控制台更新了模板中的 KS3/KPFS/Klog/Image 凭据。
- Kubernetes Secret 中的凭据没有变化。

结果：

- operator 不会从 OpenAPI 反写或更新 Secret。
- 如果用户没有修改凭据相关字段，后续 diff 请求通常不会重新提交这类敏感字段。
- 如果用户修改了挂载、镜像、日志等依赖凭据的字段，webhook 会读取当前 CR 引用的 Secret 作为请求凭据。

残余风险：

- Kubernetes Secret 可能和平台侧真实凭据不一致。
- 修改凭据相关字段时，如果 CR 缺少 `credentialRef` 或 Secret 已过期，会被 webhook 拒绝或被 OpenAPI 拒绝。

### 7.10 SandboxClaim 部分成功

场景：

- 创建 SandboxClaim 时 webhook 循环调用 `StartSandboxInstance`。
- 前几个实例启动成功，后一个调用失败。

结果：

- Admission 请求会被拒绝，Claim CR 不会创建。
- 已经启动成功的远端沙箱没有 Claim CR 关联。
- Poller 后续可能把这些沙箱 adoption 成普通 Sandbox CR。

残余风险：

- Claim 创建失败但平台侧已有部分实例，需要靠 Poller 或人工清理。

建议改进：

- Claim 创建改为 controller 异步执行，并记录每个实例的创建进度。
- 或 webhook 在失败时尝试回滚已创建实例，但这会增加 admission 延迟和失败面。

### 7.11 Operator ServiceAccount 绕过 webhook

Operator 自己更新 CR 时会被 webhook 放行，避免同步回写再次触发 OpenAPI 调用。

残余风险：

- 如果有人用 operator 的 ServiceAccount 手动提交 CR，也会绕过 webhook 校验和 OpenAPI 调用。

建议：

- 严格限制 operator ServiceAccount 的使用范围。
- 不把该 ServiceAccount 暴露给业务用户。

## 8. 当前一致性模型

当前实现采用最终一致性：

- OpenAPI 是模板、沙箱实例状态和可反推元数据的权威源。
- Webhook 负责把用户写入同步提交到 OpenAPI。
- Reconciler 负责事件驱动的快速绑定、finalizer 删除和状态汇总。
- Poller 负责控制台/SDK/OpenAPI 直接修改后的 CR 反写，以及 webhook 或 reconciler 失败后的兜底。
- Kubernetes update conflict 不作为硬失败处理，而是等待下一次同步收敛。

因此，短时间内 CR 可能出现以下临时状态：

- template-id 或 sandbox-id annotation 尚未写入。
- `spec` 已经被 OpenAPI 同步，但 `status` 尚未更新。
- 用户刚提交的字段被旧 OpenAPI 结果短暂覆盖，随后再次收敛。
- 从 OpenAPI adoption 出来的 CR 缺少凭据引用，直到用户补齐 Secret 引用。

## 9. 建议后续改进

升级已有集群时，需要先把旧 CR 的 `status.templateID` / `status.sandboxID` 迁移到对应 annotation，再应用移除 status ID 字段的新 CRD 和新 operator。否则旧 CR 会因为缺少绑定 annotation 而无法继续通过 Get OpenAPI 同步。

1. 为 CR -> OpenAPI 创建请求加入幂等键，并在 OpenAPI 响应或 List 结果中暴露该幂等键，方便 Poller 精确绑定原 CR。
2. 增加内部 annotation 绑定失败和 adoption 兜底的指标，便于观察创建链路是否频繁退化到 Poller adoption。
3. Claim 创建建议从 webhook 同步循环改为 controller 异步 reconcile，降低 admission 超时和部分成功风险。
4. 为 Poller 和 Reconciler 增加指标：同步耗时、OpenAPI 错误数、update conflict 数、adoption 数、missing remote 删除数。
5. 对持续 conflict 的 CR 增加事件或 condition，避免同步长期延迟时没有可见信号。
6. 对凭据漂移增加更明确的 status 表达，例如区分“OpenAPI 有凭据但 CR 缺少 Secret 引用”和“Secret 存在但可能与平台侧不同”。
